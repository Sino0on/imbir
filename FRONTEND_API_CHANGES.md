# Техническое руководство для фронтенда: Обновления API Imbir

Привет! В этой спецификации собраны все недавние обновления бэкенда.

---

## 1. Отзывы (Reviews)

### 1.1 Создание отзыва
Теперь при создании отзыва передаются параметры `target_type` и `target_id`.
* **Запрос**: `POST /api/reviews/`
* **Body (JSON)**:
  ```json
  {
    "target_type": "doctor", // "doctor" или "clinic"
    "target_id": 12,        // ID пользователя (врача или клиники)
    "appointment_id": 18,   // Опционально: ID записи на приём (для привязки к конкретному приёму)
    "rating": 5,
    "text": "Отличный специалист!"
  }
  ```
* **Валидация**: Отзыв к конкретной записи приёма (`appointment_id`) можно оставить только в том случае, если приём **завершён** (`status: "completed"`). При попытке оставить отзыв на незавершённый приём бэкенд вернет ошибку `400 Bad Request`.
* **Ответ (201 Created)**:
  ```json
  {
    "id": 42,
    "author": {
      "id": 3,
      "full_name": "Асанов Асан"
    },
    "target_type": "doctor",
    "target_id": 12,
    "appointment_id": 18,
    "rating": 5,
    "text": "Отличный специалист!",
    "reply": null,
    "created_at": "2026-07-01T18:00:00Z"
  }
  ```

### 1.2 Редактирование отзыва
Позволяет изменить только оценку и текст. Доступно только автору отзыва.
* **Запрос**: `PUT /api/reviews/{id}/`
* **Body (JSON)**:
  ```json
  {
    "rating": 4,
    "text": "Изменил отзыв, оценка 4."
  }
  ```
* **Ответ (200 OK)**: Возвращает обновленный объект отзыва (ReviewItem).

### 1.3 Удаление отзыва
Удаляет отзыв и автоматически пересчитывает средний рейтинг цели. Доступно только автору.
* **Запрос**: `DELETE /api/reviews/{id}/`
* **Ответ**: `204 No Content`

### 1.4 Ответ на отзыв (Врач / Клиника)
Форма ответа на `/doctor/reviews` теперь работает. Ответить на отзыв может только целевой врач или клиника.
* **Запрос**: `POST /api/reviews/{id}/reply/`
* **Body (JSON)**:
  ```json
  {
    "text": "Спасибо за ваш отзыв! Рады помочь."
  }
  ```
* **Ответ (200 OK)**: Возвращает обновленный объект отзыва с заполненным полем `reply`.

### 1.5 Поле `reply` и `target_id` во всех отзывах
Во все списки отзывов (`GET /api/reviews/`, `/api/doctor/reviews/`, `/api/clinic/reviews/`, `/api/profile/reviews/`) добавлены поля:
* `target_id` — ID цели (врача или клиники)
* `reply` — содержит объект ответа или `null`:
  ```json
  "reply": {
    "text": "Спасибо за ваш отзыв!",
    "created_at": "2026-07-01T18:15:00Z"
  } // или null, если ответа нет
  ```

## 2. Восстановление пароля

### 2.1 Запрос на восстановление
* **Запрос**: `POST /api/auth/password-reset/`
* **Body (JSON)**:
  ```json
  {
    "email": "user@mail.com"
  }
  ```
* **Ответ (200 OK)**: Всегда возвращает `200` в целях безопасности (защита от сканирования почтовых адресов). Токен отправляется на указанный email.
  ```json
  {
    "detail": "Если пользователь существует, письмо с инструкциями отправлено."
  }
  ```

### 2.2 Подтверждение сброса пароля
* **Запрос**: `POST /api/auth/password-reset/confirm/`
* **Body (JSON)**:
  ```json
  {
    "token": "строка_токена_из_письма",
    "password": "новый_пароль_минимум_8_символов"
  }
  ```
* **Ответ (200 OK)**:
  ```json
  {
    "detail": "Пароль успешно сброшен"
  }
  ```

## 3. Регистрация: Передача файлов строками (URL)

Вместо `base64` или бинарных файлов, в формах регистрации врачей и клиник теперь можно отправлять абсолютные или относительные URL-строки файлов, полученные ранее через `POST /api/upload/`.
Бэкенд поддерживает как прямую загрузку файлов, так и отправку URL-строк:
* **Регистрация Врача** (`POST /api/auth/register/doctor/`):
  * `photo` — принимает URL-строку (или файл)
  * `documents` — принимает массив URL-строк (или файлов)
* **Регистрация Клиники** (`POST /api/auth/register/clinic/`):
  * `logo` — принимает URL-строку (или файл)
  * `photos` — принимает массив URL-строк (или файлов)
  * `documents` — принимает массив URL-строк (или файлов)

---

## 4. Справочники и Фильтры (Подтверждение)

* **Подтверждено**: Вся регистрация и фильтры в каталоге принимают простые строковые названия напрямую из справочников `/api/references/*` (например, `city: "Бишкек"`, `clinic_type: "Частная"`). Никакие числовые ID или слаги передавать не нужно.

---

## 5. Проверка статуса оплаты и деталей записи на приём

* **Доступ для Врачей и Клиник**: Исправлена ошибка `403 Forbidden`. Теперь врачи и клиники имеют полный доступ к деталям (`GET /api/appointments/{id}/`) своих записей на приём.
* Вы можете опрашивать статус записи на приём по ID:
  * **Запрос**: `GET /api/appointments/{id}/`
  * **Ответ (200 OK)**: Возвращает детали записи, включая текущий статус в полях `"status"`, а также `"google_meet_link"` и `"is_online"`.

## 6. Связь Услуга ↔ Врач (Many-to-Many)

Мы изменили связь с One-to-Many на **Many-to-Many** (одна услуга может принадлежать нескольким врачам).

### Фильтрация в каталоге
* `GET /api/services/?doctor_id=12` по-прежнему работает и возвращает все услуги врача.

### Формат ответа услуг
Для обратной совместимости сохранены старые поля, но добавлены новые списки:
* **Поле `doctors` (Новое)**: Массив всех врачей этой услуги.
* **Поле `doctor` (Старое)**: Содержит первого врача из списка (или `null`).

```json
{
  "id": 1,
  "name": "Консультация терапевта",
  "category": "Терапия",
  "price": "1200.00",
  "duration": 30,
  "clinic": { "id": 2, "name": "Клиника Здоровье" },
  "doctors": [
    { "id": 12, "full_name": "Иванов И.И.", "photo": "https://..." }
  ],
  "doctor": { "id": 12, "full_name": "Иванов И.И." } // дублирует первого врача для совместимости
}
```

### Добавление услуг в кабинете врача (`POST /api/doctor/services/`)
При создании услуги врачом она автоматически привязывается к его профилю через Many-to-Many.

## 7. Новые поля приёма и заключения врача (Summary)

В модель приёма добавлены поля заключения врача: `diagnosis` (диагноз), `recommendations` (рекомендации) и `doctor_notes` (личные врачебные заметки).

### 7.1 Добавление и изменение заключения (Кабинет врача)
Врач может получить или сохранить заключение по приёму с помощью специального эндпоинта.
* **Запрос**: `GET` / `PATCH` `/api/doctor/appointments/{id}/summary/`
* **Body (JSON)**:
  ```json
  {
    "diagnosis": "Острая респираторная вирусная инфекция (ОРВИ)",
    "recommendations": "Обильное теплое питье, покой, витамин C 1000мг.",
    "doctor_notes": "Пациент жалуется на утомляемость. Прийти повторно через неделю."
  }
  ```
* **Ответ (200 OK)**: Возвращает сохраненный объект заключения.

### 7.2 Отображение в списке пациентов
В списке пациентов врача (`GET /api/doctor/patients/`) теперь отдается дополнительное поле `"diagnosis"`. Оно содержит последний установленный диагноз по завершённым приёмам этого пациента у текущего врача.

### 7.3 Отображение в истории пациента
В истории приёмов пациента (`GET /api/profile/appointments/`):
* Добавлены поля `"diagnosis"` и `"recommendations"`. Пациент видит только их (поле личных заметок врача `"doctor_notes"` скрыто).
* Добавлены поля `"is_online"` (флаг) и `"google_meet_link"` (ссылка), чтобы пациент мог повторно перейти на звонок из истории.
* **OpenAPI Фикс**: Тип поля `"can_review"` теперь возвращается строго как `boolean` (`true`/`false`), а не как строка.

## 8. Доступные слоты времени врача (Available Slots)

Бэкенд теперь вычисляет и возвращает реальное расписание и занятые слоты врача по дате.
* **Запрос**: `GET /api/doctors/{id}/available-slots/?date=YYYY-MM-DD`
  * Пример: `GET /api/doctors/5/available-slots/?date=2026-07-06`
* **Как рассчитывается**:
  1. Бэкенд находит график врача на день недели (например, `mon: 09:00 - 18:00`).
  2. Генерирует 30-минутные слоты.
  3. Маркирует слоты как `available: false`, если они пересекаются с обеденным перерывом врача (`lunch_break`), с уже существующими подтверждёнными/завершёнными записями, или если время слота уже прошло.
* **Ответ (200 OK)**:
  ```json
  {
    "date": "2026-07-06",
    "slots": [
      { "time": "09:00", "available": true },
      { "time": "09:30", "available": true },
      // ...
      { "time": "13:00", "available": false }, // обеденный перерыв
      { "time": "13:30", "available": false }, // обеденный перерыв
      // ...
      { "time": "15:00", "available": false }  // занято другим пациентом
    ]
  }
  ```

## 9. Валидация при создании записи на приём

При попытке создать запись приёма (`POST /api/appointments/`):
* Если выбранный врач или клиника неактивны или не опубликованы (`is_published: false`), запрос отклоняется с ошибкой `400 Bad Request`.
* Если выбрана онлайн-консультация (`is_online: true`), но у врача в кабинете отключена онлайн-запись (`is_online_available: false`), запрос отклоняется с ошибкой `400 Bad Request`.
